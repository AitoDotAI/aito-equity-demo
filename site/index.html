The cohorts above split on the day-1 price — but a trader wants to anticipate that move from the content, the second the release hits. So we let an LLM read each earnings press release and extract the signals a human skims for — headline tone, beat/miss framing, guidance change, EPS direction — then ask which of them actually separate the reaction. Sorted by how much each signal moves the day-1 tape.

The honest result a quant will respect: a positive headline barely moves the tape — the market already prices the consensus the company is framing against. The signal that earns its keep is the genuinely new information the release reveals — above all a change in forward guidance, which the headline buries but the engine surfaces. Naive sentiment is priced; the structural surprise is not.

Factor Explorer

Every feature — LLM judgments, sector, valuation, growth, leverage, momentum — ranked by its lift toward the great bucket (upside) and toward poor / disaster (downside). A factor strong on one side and quiet on the other is a clean signal; one that lifts both is just volatility.

exact lift = P(outcome | feature) / P(outcome), computed per feature
Feature N Lift → great (upside) Lift → poor/disaster (downside)
The two-directional read is where the interesting questions live. Airlines never reach great and tilt toward disaster. Utilities do neither — pure ballast. A wide moat lifts the upside without touching the downside. The features that move one side and not the other are the ones the market tends to misprice.

Against the alternatives

Two bars to clear: guessing the most common outcome, and the naive "throw every feature at it." Accuracy is the easy one; information gain — do the predicted probabilities actually beat the base rate? — is the honest one.

Does adding features help?

Features added in order of individual usefulness (one held-out fold). Information gain peaks around — features, then erodes — extra features stop informing and start overfitting. The demo doing model selection in the open.

The figures above are the default-inference floor that _evaluate reports. The demo's live predictions use Aito's ai: high grouping mode, which collapses correlated features into single votes — on the full 16-feature set it lifts held-out information gain further (≈ +0.04 → +0.08 measured separately), i.e. "throw every feature at it," made calibrated.

NOTES ON THE METHOD · WHAT THIS IS, AND IS NOT

On Methodology

Every line in this demo is reproducible. Every concession to imperfection is documented. The demo is the methodology; the numbers are the consequence.

№ i

Point-in-time universe

Constituents reconstructed from Wikipedia's historical S&P 500 edits as of January 1, 2014 / 2017 / 2020 — what an investor could actually have bought, not today's index. Each constituent is then tracked to its terminal state: still trading, acquired, or delisted.

№ i·b

The survivorship trap — and where we still half-fall in

The unit of analysis is a snapshot: grade a company from its filing at the vintage date, then observe what the position returned years later. A bankruptcy isn't a new data point — it's the outcome of a snapshot that already existed. The naïve study screens today's index and silently inherits the winners; we start from the point-in-time roster, so the Chesapeakes and Frontiers are present at vintage and graded from their pre-bankruptcy 10-K like any other name.

The leak is on the outcome side: our return source (yfinance) has no price history after a stock is cancelled, so a Chapter 11 that wipes out common holders returns null and would drop from every statistic — re-introducing the bias through the back door, this time hiding the losers. "Can't find the ticker → mark it gone" is the right instinct but unsafe alone: acquisitions vanish from the feed too, usually at a premium. So we resolve the reason a stock disappeared from its SEC record (Chapter 11 filing → common cancelled → booked at −100% from vintage to the filing date) and leave premium buyouts and take-unders to the price source. Scope is deliberately narrow — only confirmed equity-zeroing Chapter 11s (Frontier, Chesapeake, Windstream, Denbury, Diamond Offshore, Noble, Valaris, Rowan, Peabody). The disaster bucket now holds real −100% outcomes — but treat the absolute bankruptcy rate as a floor, not a census.

№ ii

LLM grading discipline

Qualitative features extracted from contemporary 10-K and proxy excerpts only. Every prompt instructs the model to grade from the documents alone and not to use post-vintage knowledge. All four features are graded in a single structured-output call per company; rationale strings are preserved as explainability columns.

№ iii

Lookahead stress test

The 2020 vintage is held out as a future-leakage probe. If feature lift is comparable across all three vintages, LLM lookahead bias is bounded. Reported in the calibration plot.

№ iv

Outcome bucketing

Five buckets, not regression on returns. Aligned with how investors actually think (disaster / poor / market / good / great). Yields calibrated probabilities rather than meaningless point forecasts.

№ v

Why this is not a trading signal

Twelve-year horizon, ~800 observations, no microstructure features, no transaction cost model, no consideration of capacity. This is a hypothesis test about long-horizon business quality — not an alpha generation system. The demo's value is architectural, not financial.

№ vi

The transferable lesson

The architecture — LLM-extracted qualitative features feeding a predictive database — generalises to any domain where unstructured judgment meets structured outcomes. Credit underwriting, M&A diligence, talent assessment, supplier risk. Equity research is the vehicle, not the destination.

The Problem

Most predictive systems demand a training run, a feature store, an MLOps pipeline. For equity research with hundreds of observations and dozens of features, that overhead is the entire project budget.

The Aito Approach

One endpoint, one query, calibrated probabilities and the evidence that justified them. No training, no feature engineering pipeline, no model monitoring. Load the table, ask the question.

POST /_predict
// Predict outcome bucket for a focal company { "from": "companies", "where": { "market_position": "dominant", "moat_strength": "wide", "market_quality": "secular_growth", "founder_still_ceo": true }, "predict": "outcome_bucket" }
Why it matters here
  • Calibrated, not point estimates — investors need probabilities
  • Explainable by construction — every prediction traces to evidence
  • Auditable for regulated workflows — no black box
  • Same query works at 800 rows or 8 million
Partial profiles, no retraining

A trained model needs every feature it was fitted on. Aito doesn't — it predicts from whatever you supply. Set two knobs or ten; the engine conditions on exactly what it's given and integrates over the rest. Move any grade and the forecast moves live, because nothing was pre-fit.

The Query
POST /api/query · predict
// Only the knobs you set — the rest is left out, not guessed { "from": "companies", "where": { /* your profile, whole or partial */ "moat_strength": "wide", "leadership_quality": 4 }, "predict": "outcome_bucket", "select": ["feature", "$p", "$why"], "config": { "ai": "high" } }

The same $why tree the prediction card opens: base rate × each knob's lift. Change a knob, re-query, watch the lifts recompute.

And the analogues
POST /api/query · similarity
// Real companies closest to the profile you built { "from": "companies", "similarity": { /* your profile */ }, "limit": 12 }

One built profile drives two queries — a calibrated forecast and its nearest historical precedents — with no model to train between them.

Architectural Note

Similarity, prediction, and relate are the same engine, the same index, the same endpoint family. No separate vector store, no embedding pipeline, no synchronisation problem.

A real event feed

An 8-K is corporate news the company is legally required to file — and the SEC tags it with an item code that is the theme: 2.02 earnings, 5.02 management change, 2.01 M&A, 1.01 material agreement. Free, timestamped, point-in-time.

So the reaction model needs no scraping and no labels — the regulator already labelled the news.

Why events, not attributes

A static feature (sector, moat) never happens on a day, so it can't drive a short-horizon move. An 8-K does. This is the right surface for next-day / next-week prediction.

  • Reaction at +1d / +1w / +20d, close-to-close
  • Themes from item codes — built-in, high-level, large N
  • Persistence: does the first move continue or revert?
The Aito angle

Same predictive database, new target. P(reaction | theme, company features) is one predict call — no separate event-study pipeline, no retraining when a new theme appears.

Two-directional lift

For every feature value, two numbers: lift toward great (upside) and lift toward poor / disaster (downside). Both are exact: P(outcome | feature) / P(outcome) over the companies that had that feature.

A lift of 1.0 is the base rate — no signal. Above 1 the feature concentrates that outcome; below 1 it avoids it.

Reading the table
  • High up, low down — a clean quality signal (wide moat, dominant position)
  • Low up, high down — a risk flag (airlines, high leverage)
  • Low up, low down — ballast (utilities — never great, never disaster)
  • High up, high down — just volatility, not edge
The mispricing lens

Features that were knowable at vintage yet still lift the upside — a wide moat, secular-growth market — are ones the market systematically under-weights. That gap is the Buffett thesis, quantified in both directions.

Masked held-out

_evaluate splits the data into train and test, hides the test rows' outcomes, predicts them from the rest, and scores the blind guesses. That's why these numbers are honest — the model never sees the answer it's graded on.

The Query
POST /api/v1/_evaluate
{ "test": { "eval_fold": "0" }, "evaluate": { "from": "companies", "where": { /* features via $get */ }, "predict": "outcome_bucket" } }
The engine on itself

Held-out predictions are marked correct or not and loaded into an eval_predictions table. Then a plain relate finds the features that co-occur with error — no new machinery, the same query type the Thesis view uses.

The Query
POST /api/v1/_relate
{ "from": "eval_predictions", "where": { "$on": { "prop": { "correct": false }, "on": { "true_bucket": "great" } }}, "relate": [ /* input features */ ] }
From accuracy to alpha

Same held-out predictions as the Evaluation view — but instead of scoring hit-rate, each company gets an expected outcome: Σ p(bucket) · rank(bucket) over the five probabilities Aito returns. Rank the universe by that scalar, buy the top slice, hold to today.

The ranking is leakage-safe by construction: the 5 folds are grouped on ticker, so every prediction is made by a corpus that has never seen that company.

The Score
POST /api/v1/_predict → expected rank
{ "from": "companies_train_fold_k", "where": { /* held-out company's features */ }, "predict": "outcome_bucket", "config": { "ai": "high" } } // → score = Σ p·rank; sort desc; take top-N
Two ways to open the box

Hit ? on any basket to see every pick and its realised return. Two explanations ride along, both reusing query types you've already seen:

Per pick — the $why of that company's _predict: base rate × each feature's lift. Click a Call chip — the same popup the Company Lab uses.

Per basket — a _relate over basket membership answers "what kind of company got selected here?" The same query the Error Analysis view points at its own mistakes.
